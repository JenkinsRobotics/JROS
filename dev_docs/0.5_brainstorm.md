# 0.5 brainstorm — lightweight Mochi-style face avatar

**Status:** brainstorm, not committed plan
**Date:** 2026-06-07 (operator brings Mochi reference + visual
direction 2026-06-08)
**Goal:** document the lightest viable tech stack + architecture
for a Mochi-style face avatar so we have something concrete to
review when the Mochi reference lands.

## Operator constraints (from 2026-06-07 conversation)

1. **Own the stack** — no vendoring code under restrictive
   licences; build our own implementations where needed.
2. **Performance-first** — every layer chosen for speed +
   minimum dependencies.
3. **Mochi-style face**, not Live2D VTuber.  A rich animated face
   with eyes + mouth, robot/companion identity (Vector / Cozmo /
   Eilik aesthetic).
4. **Each robot = one persona**.  No multi-persona switching.
5. **Future-compatible with hardware avatar** (same code → Jetson
   screen later).

## The lightweight tech stack

Every layer chosen with one question: *what's the smallest thing
that works?*

### Bridge: pure `websockets` library (Python ↔ browser)

Not Flask. Not FastAPI. Not Socket.IO. Just the
[`websockets`](https://websockets.readthedocs.io/) library.

| Option | Verdict |
|---|---|
| **`websockets` (pure WS, async)** | ✓ minimal, ~3000 deps install, async-native |
| Flask + Flask-SocketIO | ✗ Flask is sync; bigger install |
| FastAPI + websockets | ✗ FastAPI's value-add (routing, validation) is unused here |
| aiohttp | acceptable fallback if we need built-in static serving |
| ZMQ TCP for the renderer | ✗ JS-in-browser can't speak ZMQ cleanly |

For serving the static face renderer HTML/JS:

- **Option A**: stdlib `http.server` — zero deps
- **Option B**: bundle `aiohttp` to combine WS + static — ~one extra dep, cleaner code

Lean: **`websockets` + stdlib `http.server`** unless aiohttp's
integration simplicity wins on second look.  Either way: NO
Flask, NO FastAPI.

### Renderer: vanilla JS + SVG (no React, no Vue, no build step)

Mochi-style faces have ~5 shapes total (two eyes, mouth, maybe
eyebrows/cheeks).  **SVG with CSS transforms hits the sweet
spot:**

| Option | Lines of code (est) | Bundle size | Verdict |
|---|---|---|---|
| **SVG + vanilla JS** | ~200 | <5 KB | ✓ pick |
| HTML5 Canvas 2D | ~250 | <5 KB | acceptable alternative |
| React + Canvas / SVG | ~300 + 200 KB framework | 200+ KB | ✗ unnecessary |
| WebGL (Three.js / PixiJS) | ~400 + lib | 200+ KB | ✗ overkill |
| Live2D Cubism | ~500 + 2 MB SDK | 2+ MB + licence | ✗ wrong product |

Why SVG over Canvas for our case:
- Each face element is a DOM node; CSS transforms animate
  position/scale/rotation natively
- Easy expression swapping (change `d` attribute on a `<path>`)
- Browser GC + composition handles all rendering optimisation
- Vector → crisp at any zoom for OBS capture
- DOM inspector lets the operator debug visually

Canvas only wins if we have 100+ shapes or particle effects —
not our case.

**No build step.**  Plain HTML file you can open in any browser.
Modify, refresh, see result.  No webpack, no vite, no tsc.

### Lip sync: amplitude RMS → mouth shape (TTS chunk events)

The right data source is **the audio chunks the TTS node already
synthesizes**, not the post-playback `/sense/spoken` ack
(too late for sync).

Proposal: TTS node publishes a NEW lightweight topic during
playback:

```python
class TTSChunk(TopicMessage):
    """One chunk of synthesized speech, published as the TTS node
    enqueues it.  Carries the amplitude (RMS) so the avatar can
    drive a mouth shape from real audio energy WITHOUT having to
    receive the raw samples."""
    topic: Literal["/sense/tts_chunk"] = SENSE_TTS_CHUNK
    rms_amplitude: float = 0.0   # 0..1
    duration_ms: int = 0
    is_final: bool = False        # last chunk of the utterance
```

Avatar node subscribes; on each chunk, pushes amplitude to the
renderer; renderer interpolates mouth shape.

**Cost on the bus:** at 20 ms chunks = 50 events/sec × ~50
bytes each = 2.5 KB/s.  Negligible.

**Cost in compute:** RMS of a 20 ms float32 array is one numpy
call (~5 µs).  Lost in the noise.

### Idle animations: procedural, ~30 lines of JS

No animation library.  Pure JS timers:

```javascript
// blink — every 4-7s, eyelid Y attribute over 150ms
setInterval(() => maybeBlink(), 1000);

// breathing — sine wave on body Y position
function tick(t) {
  const breath = Math.sin(t / 2000) * 2;  // ±2px over 4s
  body.style.transform = `translateY(${breath}px)`;
  requestAnimationFrame(tick);
}

// micro-saccades — eyes shift every 2-5s
setInterval(() => maybeSaccade(), 1000);
```

The cadence numbers come from real psychology + Open-LLM-VTuber's
implementation (reference only, no code copied).

### Expression state: simple enum + interpolation

State machine, no library:

```javascript
const FACES = {
  neutral:  { eyes: 'M...', mouth: 'M...', cheeks: 0 },
  happy:    { eyes: 'M...', mouth: 'M...', cheeks: 0.5 },
  sad:      { eyes: 'M...', mouth: 'M...', cheeks: 0 },
  focused:  { eyes: 'M...', mouth: 'M...', cheeks: 0 },
  thinking: { eyes: 'M...', mouth: 'M...', cheeks: 0 },
};

function setExpression(target, durationMs = 200) {
  // Tween each property from current → target over duration
}
```

Each face is a JSON object with SVG path data for the eyes/mouth.
**Per-persona asset = one JSON file** the operator can edit by
hand or AI-generate.

## Architecture (refined for perf)

```
┌──────────────────────────────────────────────────────────────┐
│  BRAIN NODE (Mac)                                            │
│  Agent + tools + memory + persona                            │
│                                                              │
│  text_to_speech   → /act/speech                              │
│  set_avatar_state → /act/avatar_state                        │
└─────────────────┬────────────────────────────────────────────┘
                  │ in-process bus
                  │
   ┌──────────────┼─────────────────────┐
   │              │                     │
   ▼              ▼                     ▼
 STT node    TTS node               AvatarNode
              │ writes               │
              │ /sense/tts_chunk     │ SUB /sense/tts_chunk (lip sync)
              │ at 50Hz              │ SUB /sense/spoken (idle/done)
              ▼                      │ SUB /sense/transcript (listening)
        /sense/spoken                │ SUB /act/avatar_state (emotion)
                                     │
                                     ▼
                          ┌──────────────────────┐
                          │ WebSocket server      │
                          │ (Python websockets    │
                          │  library, async)      │
                          │                       │
                          │ ── HTTP at /         │
                          │     serves face.html  │
                          │ ── WS at /events     │
                          │     pushes state     │
                          └────────┬──────────────┘
                                   │
                                   ▼
                          ┌───────────────────────┐
                          │  face.html             │
                          │  (vanilla JS + SVG)   │
                          │                       │
                          │     ◉  ◉              │  ← eyes
                          │      ─                │  ← mouth
                          │                       │
                          │  Transparent bg       │
                          │  OBS captures here    │
                          └───────────────────────┘
```

Total new Python in JROS: **~250 lines** (AvatarNode + WebSocket
bridge + new TTSChunk topic + TTS node hook to publish it).

Total new JS/HTML: **~300 lines** (face.html + per-emotion SVG
paths + WebSocket handler).

Total new dependencies: **`websockets`** (one PyPI package, MIT
licensed, pure-Python, no native bindings).

## Performance budget

Targets that should be comfortable:

| Metric | Target | Why feasible |
|---|---|---|
| Bus events/sec | <100 sustained | 50 from TTS chunks + <5 from state changes |
| WebSocket throughput | <5 KB/s | tiny JSON payloads |
| Render fps | 60 sustained | DOM updates for 5-10 SVG attrs <1 ms/frame |
| RAM (Python avatar node) | <50 MB | no big buffers, no models loaded |
| RAM (browser tab) | <100 MB | single SVG element, no framework |
| Cold-start avatar | <500 ms | static HTML serve + WS connect |
| CPU at idle | <2% | procedural idle is just timers |
| CPU during speech | <5% | RMS calc + DOM tween |

For comparison: a single Live2D model + Cubism runtime + WebGL
context typically eats 100-200 MB and ~10% CPU at idle.  We come
in at maybe a tenth of that.

## Why this stack is genuinely "fast"

The "lightweight" claim is concrete:

| Layer | Choice | Alternative we rejected | Savings |
|---|---|---|---|
| Bridge | `websockets` | Flask-SocketIO | ~50 deps |
| Bridge | `websockets` | FastAPI | ~30 deps |
| Renderer | SVG + vanilla JS | React | ~200 KB + build step |
| Renderer | SVG + vanilla JS | Live2D | ~2 MB + licence complexity |
| Animation | procedural CSS/JS | GSAP / Anime.js | ~50 KB |
| Lip sync | RMS in TTS node | client-side audio analysis | network round-trips |
| Build | none | webpack / vite | toolchain |

Total runtime weight of the avatar subsystem:
- Python: AvatarNode + websockets ≈ 5 MB resident
- Browser: one SVG element + ~10 KB JS ≈ 80 MB browser tab (mostly Chrome overhead)

Same as adding a small chat window to JROS.

## Closing — what's worth pushing back on if you disagree

- **No build step for the renderer.**  Hard line.  If you want
  fancy animation libraries later, fine, but for MVP a plain
  HTML file you can open in any browser is dev velocity gold.
- **TTSChunk topic is new infra in topics.py.**  Adds an msgspec
  class, registers in TOPIC_TO_CLASS, codec classification.
  Small but non-zero change to the 0.4-shipped SSOT.  Worth it
  because lip sync needs amplitude data and trying to retrofit
  it later is painful.
- **`websockets` is one new dependency.**  Could in theory use
  stdlib `socket` + hand-rolled WS handshake, but that's masochism
  for the savings.  One small dep is the right tradeoff.

If anything here feels wrong, push back tomorrow when we sync
on the Mochi reference.
