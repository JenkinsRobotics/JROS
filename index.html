// what it is

A real-hardware agentic OS, not a chat wrapper.

JROS is the operating framework for Jaegers — autonomous agents that share a single coherent runtime regardless of body. A Jaeger might be a desktop companion today, an LED-faced drone next quarter, or a humanoid arm next year — all running the same agent loop, memory, skill registry, and tool surface. The brain doesn't fork when you change the body.

Built from real hardware pain on Apple Silicon and Jetson Orin. No Docker, no special OS versions, no dependency hell. git clone simple, with one curl line for the rest.

Local-first by default. The agent runs on your hardware against your models. No cloud account required. No usage caps. No telemetry. If you want to bring an OpenAI key or a Claude key, it's one config line — but you don't need one.

// the framing

A sound soul, dwells within a sound mind, and a sound body.

A robot isn't one program. It's three swappable pillars composed in one framework — a Mind to reason, a Body to act on the world, and a Soul to be someone. JROS lets you mix them freely.

Mind

The agentic agent — runners, tools, memory, reasoning. The intelligence that plans and decides. Swap the mind that drives the robot.

agent · the Hermes side

Body

The middleware hardware that controls real-world tools — motors, cameras, LEDs, mics — through nodes, adapters, capabilities, and fail-closed safety. Swap the body: humanoid today, drone tomorrow.

hardware · the ROS 2 side

Soul

The persona — identity, character, voice, and lore, growing more advanced over time. Who the agent is, independent of the body it wears or the mind that drives it.

persona · character

Hermes owns the Mind. ROS 2 owns the Body. JROS is the framework where a Mind drives a Body wearing a Soul — all three modular, all three yours to compose.

// capabilities

Everything an agent needs to live in the real world.

~100 built-in tools plus 99 self-authored skills. A CORE set is always visible; the rest unlock on demand via describe_tool / load_toolset.

Local LLM

In-process Gemma-4 via llama.cpp on Metal — E4B for real-time, 26B-A4B for Deep Think. Dual-context inference lanes keep the conversation warm: persona-on cold start 45.6s → 0.71s.

gemma-4 · metal · dual-lane

Voice — production-grade

Wake-word required by default ("ok jaeger" + phonetic variants). 10s follow-up window. AEC echo cancellation for video-call style barge-in. Kokoro TTS + Whisper STT.

wake · AEC · barge-in

Tool surface

Files, memory, web, code execution, scheduling, kanban, delegation, computer use. MCP servers plug in as first-class peers. Every action is permission-gated (confirm / allow) and audit-logged.

~100 tools · MCP

Memory — semantic + episodic

The agent remembers facts about you, the work you do, and how the previous conversation ended. Search by meaning, not just keyword. Persistent across sessions.

facts · episodes · search

Kanban + Deep Think

Multi-step work decomposes into kanban cards the agent self-promotes. An idle "sleep cycle" swaps to a heavier coder model and drains a skill-development queue.

/board · deepthink

Computer use

The flagship macOS skill: drive any app through the accessibility tree. See the screen. Click, type, traverse menus. Auto-grants escalating permission as the agent earns it.

macOS · accessibility

Self-improving skills

The agent writes, smoke-tests, benchmarks, and versions its own skills — then improves them. After each use it logs a structured note; on idle it audits its own trajectory in the second person ("review your own work as if someone else's"), distils one rule, and keeps the rewrite only if it measurably beats the old one. Superseded versions are archived; skills that never win retire.

notes · 2nd-person audit · keep-if-better

Native Mac app

Since 0.7, JaegerOS.app is the primary UI — menu-bar resident, splash → chat / settings windows, quit-from-tray. It talks to the Python core over a versioned NDJSON bridge: usable in ~0.5s while the model warms behind it. The terminal (--tui) stays first-class for SSH / headless.

menu-bar · bridge v1 · TUI

Embodiment-ready

Body contract + capability-gated skill loader in place. Same agent loop on a Mac today, on a Jetson Orin tomorrow. The brain doesn't care what the body is.

Mac · Jetson · ESP32

Two-runner agentic core

A real-time runner — a fluid loop with a hard boundary: a verify gate that catches a stalled plan or a claim-without-action, and a persona filter that restores character voice with no reasoning tax. Plus a staged plan → execute → verify Deep Think runner for the heavy jobs.

verify gate · persona filter

Your agent, its own identity

Name your robot Ted — it can play HAL. The agent's name and profile picture belong to the instance and never change with the persona it wears. Swap characters freely; the identity you built stays put.

name · picture · persona

Benchmarked + security-gated

Every pipeline change holds a routing corpus at 79/81, and a 51-case hermetic scenario suite runs the whole system end-to-end against a throwaway instance — including 15 security gates (prompt injection, credential-sweep, memory-poisoning). Safety is a measured gate, not a promise.

79/81 · 51 scenarios · 15 sec gates

// architecture

Two clear buckets. Sibling dirs. Nothing scattered.

Every JROS install lives in one directory. The framework is git-tracked; your data is gitignored. Upgrades are git pull. No ~/.config dotdirs, no /usr/local sprinkles, no site-packages confusion.

~/jaeger/jaeger_os/

Framework code

The brain: agent loop, tool surface, memory engine, skill registry, prompt assembly. Upgraded in place by jaeger update. Never holds your personal state.

~/jaeger/.jaeger_os/

Operator state

Everything yours. instances/<name>/ per agent — persona, config, memory, skills, prompts, workspace, logs, credentials. Fully .gitignored. Upgrades never touch it.

Runtime model

The brain shipped first. 0.4 wired the spine.

jaeger boots the agent: one Gemma instance, the persistent voice pipeline, the skill registry, the memory store — all in one Python process for sub-microsecond function-call latency. 0.4 adopted a node architecture: brain stays one process; audio I/O, STT, TTS, vision, motors, lights each become their own node, talking over the ZmqBus. Same code laptop or fleet — only the transport changes. (0.5 added the character system; 0.6 wrapped it in a real install/update/lifecycle shell; 0.7 made it a native Mac app with a measured two-runner agent core.)

0.4 node layout

One brain process. N hardware-bound peripheral nodes.

Tools = networking shims; nodes = execution. The agent's tool signatures (text_to_speech("hi"), listen(seconds=5)) stay identical — what changes is the implementation: in-process call becomes a topic publish + correlation-ID wait for the ack. STT and TTS get their own nodes because voice pipelines evolve (today's Kokoro becomes tomorrow's MLX-TTS without touching the brain). The brain doesn't know whether /sense/audio_in comes from a Mac mic, a Jetson mic, or a simulated WAV file.

Architecture + the two runners in agentic_runners.md.

// where jros fits

The framework no one else owns.

ROS is the embodied-robotics middleware standard but doesn't think about LLM agents. Hermes and the agent frameworks own the agent loop but assume a single process on one machine — no robots in the picture. JROS is built where they don't overlap: a local LLM agent that drives real hardware nodes.

// lifecycle

Behaves like real software — install, run, update, repeat.

One jaeger command for the whole life of a unit — no git on the box, no PyPI, no re-pasting the curl line. Your agents and their memory survive every update.

operate — bash 
# manage agents — each plays a character
$ jaeger agent create|list|use|delete|clear

# run it — manually, or unattended after a reboot
$ jaeger                  # opens JaegerOS.app · --tui for the terminal
$ jaeger autostart enable
$ jaeger dev               # the dev app, pinned to the jros-dev instance

# keep it current — downloads + applies in place
$ jaeger doctor            # "0.7.x available"
$ jaeger update            # --rollback · --ref to pin

# recover or remove — agents kept unless --purge
$ jaeger reinstall
$ jaeger uninstall

Updates download the release and swap the product in place — your .venv and every agent's memory, skills, and credentials are never touched. --rollback reverts to the previous version, and the tray + Jaeger Studio surface "update available" on their own.

// roadmap

Brain first. Body next. Same agent loop throughout.

0.1–0.3 — Shipped

The agent layer

Local-first agent, self-authored skills, kanban, Deep Think, the external-model pipeline, computer_use — then the System/Runtime/User layout, voice (wake-word + AEC barge-in), messaging bridges, the tray, and the skill-v3 system.

0.4.0 — Shipped

Node architecture

The ZmqBus + body contract. The brain stays one process; audio I/O, STT, TTS, vision, motors, lights become bus-addressable nodes behind clean adapter protocols. The bones embodiment needs.

0.5.0 — Shipped

Character system + Studio

Transport-agnostic voice, the character system (each agent plays a character), the animation/avatar pipeline, the skill tree, and the Jaeger Studio GUI (since spun out to its own repo).

0.6.0 — Shipped

The product shell

Feels like real software: a clean install, in-place jaeger update with rollback, autostart, reinstall / uninstall — plus the agentic-quality arc (autonomy modes, memory provenance, a self-improving skill loop).

0.7.0 — Shipped

Swift-first + the two-runner core

The native Mac app (JaegerOS.app) over a versioned NDJSON bridge; the measured real-time + Deep Think runners with dual-context inference lanes; the agent-name / profile-picture split from the character; and a hermetic scenario benchmark alongside the routing corpus.

0.8 — Next

The modular framework

Modules — TTS / STT / vision / hardware nodes, plugins — that own their own config, settings, and lifecycle and register into one unified surface. The ROS2-style federation seam that turns JROS from one agent into a framework for composing many.

Later

Hardware · JP01

The first hardware Jaeger — the same brain Lilith runs, a new body around it. Mic-in / RGB-out / speaker-out streaming on the device over the MediaFrame / ZmqBus seam.
